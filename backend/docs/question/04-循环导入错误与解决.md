# 循环导入错误与解决

## 问题描述

启动应用时出现循环导入错误：

```
ImportError: cannot import name 'scheduler' from partially initialized module 'config.get_scheduler'
(most likely due to a circular import)
```

## 错误原因分析

### 循环导入链路

```
config/get_scheduler.py (第22行)
  → 导入 register_daily_refresh_scheduler
  → daily_refresh_scheduler.py (第13行)
    → 导入 scheduler
    → config/get_scheduler.py (循环！)
```

### 导入关系图

```
┌─────────────────────────────┐
│   config/get_scheduler.py   │
│                             │
│   from daily_refresh_       │
│   scheduler import          │
│   register_daily_...        │
│                             │
│   scheduler = AsyncIOSch... │ ◄───┐
└─────────────────────────────┘     │
          │                         │
          │ 导入                     │
          ▼                         │
┌─────────────────────────────┐     │
│ daily_refresh_scheduler.py  │     │
│                             │     │
│ from config.get_scheduler   │─────┘
│ import scheduler            │
│                             │
│ register_daily_refresh_     │
│ scheduler()                 │
└─────────────────────────────┘
```

### 代码示例

**config/get_scheduler.py:**
```python
from module_task.scheduler.daily_refresh_scheduler import register_daily_refresh_scheduler

scheduler = AsyncIOScheduler()
# ... 配置调度器 ...

def init_system_scheduler():
    register_daily_refresh_scheduler()  # 在这里调用
```

**daily_refresh_scheduler.py (修改前):**
```python
from config.get_scheduler import scheduler  # ← 模块顶层导入，导致循环

def register_daily_refresh_scheduler():
    if not scheduler.running:
        # ...
```

## 解决方案：延迟导入

### 原理

将 `scheduler` 的导入从**模块顶层**移到**函数内部**，使其在实际使用时才导入。

### 为什么延迟导入有效？

1. 当 `get_scheduler.py` 导入 `daily_refresh_scheduler.py` 时，模块开始加载
2. 但由于 `scheduler` 的导入在函数内部，**模块顶层不会触发对 `get_scheduler` 的回导入**
3. `daily_refresh_scheduler.py` 模块加载完成
4. 控制权回到 `get_scheduler.py`，此时 `scheduler` 已经定义
5. 后续调用 `register_daily_refresh_scheduler()` 时，函数内部的导入才会执行

### 修改后的代码

**daily_refresh_scheduler.py (修改后):**
```python
# 模块顶层不再导入 scheduler
# from config.get_scheduler import scheduler  # ← 删除这行

def register_daily_refresh_scheduler():
    # 延迟导入，避免循环依赖
    from config.get_scheduler import scheduler

    if not scheduler.running:
        # ...

def unregister_daily_refresh_scheduler():
    # 延迟导入，避免循环依赖
    from config.get_scheduler import scheduler

    # ...
```

## 适用场景

延迟导入适用于：

| 场景 | 是否适用 |
|------|----------|
| 模块顶层导入，函数内使用 | ✅ 适用 |
| 仅用于类型注解 | ✅ 适用（或使用 `TYPE_CHECKING`） |
| 模块顶层就使用（如装饰器） | ❌ 不适用 |

## 其他解决方法

### 方法1：合并模块
将 `register_daily_refresh_scheduler` 移到 `get_scheduler.py` 中。

### 方法2：使用 TYPE_CHECKING
如果只用于类型注解：
```python
from typing import TYPE_CHECKING
if TYPE_CHECKING:
    from config.get_scheduler import AsyncIOScheduler
```

### 方法3：反转依赖
让 `get_scheduler` 在初始化后调用注册函数，而不是在导入时执行。

## 总结

- **循环导入本质**：A 导入 B，B 又导入 A
- **延迟导入原理**：推迟导入时机，打破循环链
- **适用场景**：依赖项在函数内部使用，不在模块顶层使用
- **最佳实践**：设计模块时应尽量避免相互依赖
